# Industrial Processing System

Thread-safe producer-consumer job processing system built with C# and .NET 8.

It supports:
- concurrent producers and worker consumers
- priority-based scheduling (lower number = higher priority)
- idempotent submission by `Job.Id`
- retry with timeout and final abort behavior
- event-based file logging
- periodic XML processing reports
- XML-based startup configuration

## Tech Stack

- .NET SDK 8.0 (see `global.json`)
- C# (`net8.0`)
- xUnit for tests

## Project Layout

- `IndustrialProcessingSystem/` main console app and processing engine
- `IndustrialProcessingSystem/Configuration/SystemConfig.xml` startup configuration
- `IndustrialProcessingSystem.Tests/` automated tests

## Quick Start

```bash
dotnet restore
dotnet build
dotnet run --project IndustrialProcessingSystem
```

When running:
- worker tasks start based on `WorkerCount`
- producers continuously submit random jobs
- stop with `Enter` or `Ctrl+C`

## Run Tests

```bash
dotnet test
```

## Configuration

The app loads `Configuration/SystemConfig.xml` from the application output directory (`bin/.../net8.0/Configuration/SystemConfig.xml`).
Edit the source config at `IndustrialProcessingSystem/Configuration/SystemConfig.xml`; it is copied on build/run.

Example:

```xml
<SystemConfig>
  <WorkerCount>5</WorkerCount>
  <MaxQueueSize>100</MaxQueueSize>
  <Jobs>
    <Job Type="Prime" Payload="numbers:10_000,threads:3" Priority="1" />
    <Job Type="IO" Payload="delay:1_000" Priority="3" />
  </Jobs>
</SystemConfig>
```

`Job` attributes:
- `Type`: `Prime` or `IO`
- `Payload`: job-specific payload format
- `Priority`: integer > 0 (lower number processed first)
- `Id` (optional): GUID; autogenerated if omitted

## Job Payloads

- `Prime`: `numbers:<value>,threads:<value>`
- `numbers` must be > 1
- `threads` is clamped to `1..8`
- result is the count of prime numbers in range `2..numbers`

- `IO`: `delay:<milliseconds>`
- delay must be an integer >= 0
- result is a random integer `0..100` after simulated delay

## Runtime Behavior

- each job has up to 3 attempts
- each attempt has a 2-second timeout
- non-final failures raise `JobFailed`
- final failure raises `JobAborted`
- successful execution raises `JobCompleted`

The system capacity is enforced by `MaxQueueSize` across queued + in-flight jobs.

## Generated Artifacts

- `logs/job-events.log`  
  asynchronous event log (`COMPLETED`, `FAILED`, `ABORT`)
- `reports/report-0.xml` ... `reports/report-9.xml`  
  periodic XML report files every minute with rotating retention (10 slots)
